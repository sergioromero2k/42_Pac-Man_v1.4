02 — FLUJO DE GIT

Protocolo de ramas

Rama por tarea, no por persona. Nada de "rama-A" / "rama-B" fijas donde cada uno acumula días de trabajo — eso genera un merge gigante al final, con conflictos grandes y difíciles de resolver.

main # siempre estable, funciona con Makefile run └── dev # rama de integración diaria ├── feature/config-parser (A) ├── feature/maze-loader (A) ├── feature/player (A) ├── feature/ghost-ai (A) ├── feature/highscore (A) ├── feature/cheat-mode (A) ├── feature/renderer (B) ├── feature/ui-menu (B) ├── feature/ui-hud (B) ├── feature/ui-pause-gameover (B) └── feature/packaging (B)
  • main solo se actualiza vía merge desde dev, nunca directo.
  • Cada tarea nueva = rama nueva desde dev — nunca trabajéis los dos en la misma rama a la vez.
  • Commits pequeños, prefijo por tipo: feat: fix: test: docs: refactor: (Conventional Commits) — facilita el changelog para la documentación de gestión.
  • Pull Request obligatorio para mergear a dev, aunque seáis dos — el otro revisa antes de aprobar. Doble uso: evidencia de "peer review" para la gestión del proyecto.
  • git pull --rebase antes de empezar cada día, para evitar historiales divergentes.
  • Borrar la rama feature en cuanto se mergea, para mantener el repo limpio.
daily-flow.sh
# Al empezar una tarea nueva
git checkout dev
git pull --rebase
git checkout -b feature/nombre-tarea

# Mientras trabajas (commits pequeños y frecuentes)
git add archivo.py
git commit -m "feat: descripción corta de lo que hace"

# Al terminar la tarea → PR hacia dev → revisión → merge
git push origin feature/nombre-tarea
git checkout dev && git pull --rebase
git branch -d feature/nombre-tarea
git push origin --delete feature/nombre-tarea

# Traer cambios de dev a tu rama activa
git checkout feature/nombre-tarea
git rebase dev

# Revisión rápida antes de mergear
git diff dev...feature/nombre-tarea

Resolviendo un conflicto de merge

  1. Git avisa qué archivo(s) tienen conflicto.
  2. Abrís el archivo y buscáis <<<<<<< ======= >>>>>>>.
  3. Decidís juntos (chat/llamada de 2 min) qué versión se queda, o combináis ambas.
  4. Borráis las marcas, guardáis, git add sobre ese archivo.
  5. git rebase --continue (o git commit si era un merge normal).
  6. Corréis tests/lint de nuevo antes de pushear, por si el conflicto rompió algo silenciosamente.
🔧 Extra recomendado: hook de pre-commit (flake8 + mypy) para que nunca llegue código roto a dev. Y un tablero Kanban (GitHub Projects / Trello / Notion) — os sirve de organización real y de evidencia para el apartado "Project management".

03 — CRONOGRAMA

Línea de misión

20 entradas de bitácora. Los días marcados SYNC no son opcionales: si os saltáis la sincronización de interfaces, ahí es cuando aparecen los conflictos de Git.

Fase 0 — Kickoff

01 Kickoff — JUNTOS Joint

Definir interfaces/contratos: qué métodos tiene Ghost, qué devuelve MazeLoader.load(), forma del config ya parseado. Escribirlo como stubs con type hints.

Teoría: OOP (composición vs herencia) · diseño por contrato · flujo de branches en Git.

04 — HÁBITOS

Tácticas extra

Cuatro rutinas pequeñas que ahorran dolores de cabeza grandes.

◆

Daily de 5–10 min

Qué hice ayer, qué haré hoy, qué me bloquea. No hace falta que sea presencial ni largo — mantiene sincronizados los contratos entre módulos.

✓

Checklist pre-merge

¿Pasa flake8? ¿pasa mypy? ¿tiene docstrings? ¿tiene al menos un test si es lógica del core? ¿rompe algo en dev?

◈

DECISIONS.md

Documento vivo de decisiones técnicas: cada elección importante (ej. "heurística greedy en vez de A* para los fantasmas") anotada con una línea del porqué.

▣

Registro de bloqueos

Anotad los bloqueos reales que surjan (ej. "el paquete A-Maze-ing no traía tal método") — contenido directo para "Summary of any blocking points".

05 — GESTIÓN DE RIESGOS

Riesgos y mitigación

Punto de partida rápido para el análisis de riesgos que pide el apartado de gestión del proyecto.

RiesgoSeveridadMitigación
Conflictos de merge grandes al finalAltaRamas por tarea (no por persona), commits pequeños, avisar en chat antes de tocar archivos compartidos.
Contratos de interfaz desincronizados entre A y BAltaRespetar los días de Sync marcados (05 y 09) y el checkpoint del Día 10.
El paquete A‑Maze‑ing asignado falla o tiene una interfaz inesperadaMediaAdaptador (maze_loader.py) que aísla el resto del código; manejo de errores sin crash desde el Día 3.
Retraso general de calendarioMediaDía 10 como punto de control natural para replanificar el resto de las tareas.
No poder explicar el módulo del compañero en la defensaMediaDía 20 de repaso cruzado + PRs revisadas por el otro durante todo el proyecto.

Notas clave

  • Los días de "Sync" no son opcionales. Si os saltáis la sincronización de interfaces, es cuando aparecen los conflictos de Git.
  • Si vais con retraso, el Día 10 (integración) es el punto de control natural para replanificar el resto.
  • No escribáis vuestro propio generador de laberintos — está explícitamente prohibido en el subject.
